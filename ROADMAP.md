## В планах

### 🔭 Крупная идея (не спланирована, ждёт фидбека продакта/инженера) — единый сервис вместо JobScraper + ResumeBuilder + Notion

**Зафиксировано пользователем:** 2026-07-02.

**Идея:** объединить поиск вакансий (JobScraper), создание резюме (ResumeBuilder) и анализ/оценку вакансий в один сервис с единым фронтендом — Notion как основной UI-слой перестал справляться с текущим объёмом и сложностью процесса.

**Три сценария использования, которые должны продолжать работать:**
1. **Скрапер как есть:** ищет вакансии, создаёт карточки, оценивает их (ATS score). Пользователь приходит и повторно оценивает уже созданные карточки вручную; если подходит — двигает в дальнейший статус. Повторная ручная оценка сейчас физически происходит в другой части системы — ResumeBuilder.
2. **Ручной поиск:** пользователь сам находит вакансии (в основном по Израилю и на LinkedIn), сам оценивает, и если подходит — подаётся (это тоже создаёт карточку по ходу дела).
3. **Движение по воронке:** если пришло приглашение на интервью — карточка двигается по канбан-доске на следующий статус.

**Конкретные проблемы с текущей схемой (Notion + два отдельных проекта):**
- Нет стандартизированных причин отклонения карточки → нельзя отслеживать качество работы скрапера (пересекается с уже запланированным пунктом "Категории причин отклонения" ниже, но эта идея шире — про саму архитектуру, не только про одно select-поле)
- Нет причины отклонения "вакансия закрыта" → риск повторной подачи в ту же компанию на ту же/похожую позицию
- При повторной проверке карточек, найденных скрапером (когда пользователя не устраивает автоматическая оценка) — риск создания дублирующей карточки в общей базе, потому что повторная оценка происходит в другой части системы (ResumeBuilder), не имеющей прямой связи с оригинальной записью
- Плохо отслеживается сквозная статистика — не видно ни основных причин отклонения резюме, ни общей картины воронки в одном месте

**Связанные уже запланированные пункты (эта идея, вероятно, поглощает или переопределяет их архитектурно):**
- "ResumeBuilder ↔ DB интеграция + status history" (см. выше, п.2 высокого приоритета) — уже решает часть проблемы дублирования карточек через `db_manual.py`/`sync_notion.py`, но остаётся внутри модели "Notion как фронтенд"; эта идея ставит под вопрос саму модель
- "Категории причин отклонения вакансии" (среднего приоритета ниже) — частный случай более широкой проблемы стандартизации отклонений

**Статус:** идея зафиксирована, архитектурное решение не принято. Нужен отдельный раунд обсуждения с продактом и инженером/архитектором прежде чем проектировать — см. фидбек ниже (запрошен 2026-07-02).

**Фидбек продакта и архитектора (2026-07-02) — консенсус: НЕ строить единый сервис/фронтенд сейчас.**

Оба сошлись: все 4 названные проблемы — это баги модели данных и dedup-логики, не ограничения платформы Notion. Кастомный фронтенд не решает ни одну из них сам по себе (некому "изобрести" словарь причин отклонения или dedup-ключ на UI-слое — это бэкенд).

**Продакт:** цена кастомного фронтенда для соло-проекта — ~70–90 часов разработки + 10–15 ч/мес поддержки навсегда (auth, канбан, мобильная адаптация, хостинг) против того, что Notion уже даёт бесплатно (мобильное приложение, канбан, фильтры, zero maintenance). Матрица не сходится при текущем объёме боли. Рекомендованный лёгкий фикс — ~10–15 часов: enum-поле причины отклонения (включая "вакансия закрыта"), dedup-ключ, доработка уже существующего дашборда (funnel-график, top rejection reasons, % qualified→applied как метрика качества скрапера).

**Архитектор — важная находка, меняющая расклад:** у нас **уже есть кастомный фронтенд** — `dashboard.py`, задеплоенный FastAPI+Chart.js сервис на Railway. "Построить единый сервис" на практике = "сделать уже существующий read-only дашборд read-write и перенести write-путь с Notion на Postgres" — это не гринфилд-проект, а расширение того, что уже работает.

Разбор проблем по корневой причине — все 4 решаются в текущей архитектуре:
- Причины отклонения / "вакансия закрыта" → недостающий enum, `config.py` + Notion single-select. Чистый data model fix.
- Риск повторной подачи → отдельная guard-проверка в `db_manual.py` (SELECT по company+normalized title на `status = position_closed` перед созданием карточки) — это единственное, что Notion действительно не может (no join-before-write), но решается 10 строками кода в бэкенде, не новым UI.
- Сквозная статистика → это не UI-проблема, а проблема неполноты данных: как только появится `status_log` + enum, funnel — это несколько `GROUP BY` запросов поверх уже существующего дашборда.

**Критическая находка про дубликаты карточек:** уже спроектированный план (`notion_id` + merge-by-url в `db_manual.py`) дедуплицирует **Postgres-строку**, но НЕ предотвращает создание второй Notion-карточки — потому что карточку создаёт отдельная, независимая логика в ResumeBuilder Step 2.5, которая не знает про `db_manual.py`. Merge задним числом не убирает уже созданный дубль-артефакт, который видит пользователь. **Fix:** dedup-проверку нужно перенести на момент создания карточки в ResumeBuilder Step 2.5 (проверять по URL существующий `notion_id` до создания, при совпадении — UPDATE, не INSERT), плюс URL как ключ матчинга слишком хрупкий (LinkedIn/ATS/company-page ссылки на одну вакансию не совпадут дословно, плюс tracking-параметры) — нужна нормализация URL + fallback на `company + normalized_title`, тот же класс проблемы, что уже отмечен для `_company_match()` на Lever/Ashby.

**Рекомендованная последовательность (оба согласны — чинить данные до фронтенда, не наоборот):**
1. Status-vocabulary enum в `config.py` + rejection-reason single-select в Notion (включая `position_closed`)
2. `notion_id` + перенос dedup-проверки на момент создания карточки в ResumeBuilder (не только в `db_manual.py`) + нормализация URL для матчинга
3. Reapplication-guard запрос в `db_manual.py` против Postgres
4. `status_log` + `sync_notion.py` (уже спроектировано выше) + funnel-графики в существующем дашборде

**Если после нескольких недель первопричина всё ещё "Notion как write-поверхность неудобен" (не "мне не хватает данных"):** архитектор рекомендует расширить существующий `dashboard.py` через HTMX (server-rendered HTML-фрагменты, один язык, один деплой, без build-тулчейна) — не React/Vue/SPA (лишний build-toolchain, сложнее для AI-парного программирования) и не Retool (отдельный вендор, конфиг вне git). Самая маленькая реальная версия — `sqladmin`, накрученный на тот же FastAPI-сервис — CRUD-таблица почти без кода за один вечер; HTMX-канбан — только если таблица реально начнёт раздражать.

**Пересмотрено пользователем 2026-07-02 — решение: строим кастомный фронтенд, не откладываем.** Три аргумента, которые перевесили рекомендацию "чинить данные, потом переоценить":
1. **Асинхронность — реальная архитектурная дыра, не мелочь.** Если Notion остаётся write-поверхностью с синком 1-2×/день в Postgres, reapplication guard (защита от повторной подачи в компанию с только что закрытой позицией) будет проверять устаревшие данные в окне между действием пользователя и синком. То же самое ломает саму цель "сквозной статистики" — она систематически на день позади.
2. **Дублирование модели данных.** План с enum'ами требует завести поле в Notion (UI) + маппинг в `config.py` + логику синка — модель данных продумывается дважды. Дешевле спроектировать её один раз напрямую в своей БД.
3. **Портфолио-ценность как отдельный, независимый драйвер.** Не только "экономия трения", но и демонстрируемый builder-кейс для интервью — меняет расчёт "стоит ли 70-90 часов" в другую сторону, т.к. окупаемость не завязана только на 4 названные проблемы.

**Rollout — поэтапно (решение пользователя):** новый фронтенд сначала берёт на себя только сценарии 1 и 3 (ручной пересмотр карточек скрапера, канбан-движение по статусам, причины отклонения, статистика). Скрапер продолжает писать в Notion как раньше, пока новый UI не докажет себя — ниже риск, можно откатиться. Postgres становится источником правды именно для этой новой части процесса сразу, без промежуточного синка через Notion. Полный вывод Notion из процесса (сценарий 2, ResumeBuilder) — отдельное решение позже.

Техническая архитектура — следующий шаг, планируется отдельно.

Архитектура:
1. Рефакторинг блоков Greenhouse/Lever/Ashby в `utils.py` (сейчас заточены под enrichment одной вакансии) в переиспользуемые `list_greenhouse(slug)` / `list_lever(slug)` / `list_ashby(slug)`, возвращающие **все** постинги компании — дальше через существующий `passes_role_filter` отбираются PM-вакансии

3. Список компаний — в Postgres-таблице `target_companies(slug, name, ats, status, source, added_at)`, засеянной из env var (по аналогии с `TELEGRAM_JOB_CHANNELS`), а не хардкод в коде — потому что список должен расти
4. Discovery новых компаний — **weekly, не daily** джоба, кандидаты пишутся в `target_companies` со статусом `pending`, активация — вручную (review queue), никогда авто-промоушен

QA — найден баг, актуальный уже сейчас (не зависит от этой фичи): **`_company_match()` в `utils.py` применяется только к Greenhouse-ветке (строка ~91); Lever (~106) и Ashby (~119) принимают первый совпавший slug без проверки имени компании** → уже сейчас возможны ложные совпадения (например, slug случайно попадёт на вакансию другой компании с похожим названием). Чинить отдельно, до/независимо от EdTech-фичи — см. Backlog.

QA-требования к фиче при реализации:
- Схема компании: `source` (manual/LLM/web-search), `date_added`, `last_verified_date`, `status` (active/dead/ats-unknown)
- Двухэтапный discovery: LLM/поиск генерирует кандидатов → код детерминированно проверяет (домен резолвится, ATS-slug проходит `_company_match()`) → никогда не доверять LLM-выводу напрямую
- Weekly re-validation sweep существующих компаний (ATS ещё жив? имя ещё матчится?) — деградировать в `dormant`, не удалять молча
- Sanity-алерт: 0 вакансий 2+ прогона подряд ИЛИ всплеск >3x baseline (похоже на slug collision)
- Дедуп против уже существующих источников переиспользовать, не городить параллельный

Порядок работ:
1. Рефакторинг `utils.py` ATS-блоков в `list_*(slug)` + фикс `_company_match()` на Lever/Ashby (high/S)
2. `target_companies` таблица + env-var seed (high/S)
3. Ручная курация 50 EdTech/LMS компаний (high/S, данные не код — Gartner/G2 LMS category/LinkedIn)
4. `sources/company_direct.py` читает active-компании из БД (high/M)
5. Weekly discovery job → pending queue + ручная активация (medium/M)
6. Через 2 месяца — замерить сигнал, решить, стоило ли того

- **Слой 1 (4×/день, синхронно с основным cron):** ВСЕ ~33 подтверждённые израильские компании (после чистки списка) + 8 глобальных с высокой уверенностью по ATS (Docebo, Cornerstone, Absorb, Instructure, D2L, Moodle HQ, SAP/Litmos, Adobe)
- **Слой 2 (1×/день, отдельный job):** остальные ~25–30 глобальных компаний среднего размера с разумным hiring-сигналом
- **Слой 3 (fallback, 2–3×/неделю или полностью через существующие источники):** нишевые/маленькие компании с неподтверждённым наймом — не строить для них отдельный polling, полагаться на Telegram/агрегаторы

*"Не терять вакансии" — carier-page fallback (добавлено по явному запросу пользователя, продакт изначально не рекомендовал из-за ROI):*
Реалистичное покрытие через структурированный ATS (Greenhouse/Lever/Ashby/join.com) — **~53% компаний**. Для остальных — generic-скрапер карьерных страниц (`/careers`, `/jobs`), переиспользующий уже существующий паттерн `_expand_listing_page()` из `sources/telegram_channels.py` (заходит на listing-страницу, тянет все ссылки, фильтрует по PM-ключевым словам). Работает для server-rendered страниц; JS-SPA (React/Vue) не парсится без headless-браузера (Playwright) — это отдельная, более тяжёлая задача, не в этом этапе.

*"Подаваться быстро" — instant-путь для Слоя 1:*
Вакансия с флагом `target_company_match=True` (Слой 1) и score ≥ 60 → создание Notion-карточки сразу (не ждать батч 4×/день) + отдельное мгновенное Telegram-сообщение (не общая сводка): "🎯 [Компания] | [Роль] | Score: XX/100 | [прямая ссылка]". ResumeBuilder Step 2.5 находит карточку уже созданной → экономит шаг.

*Фазы:*
1. Рефакторинг `utils.py` в `list_*(slug)` + фикс `_company_match()` на Lever/Ashby + таблица `target_companies` + seed чищенным/расширенным списком (Слой 1 сразу активен, остальные — `pending`)
2. `sources/company_direct.py` для Слоя 1 (4×/день) + instant-alert путь
3. Через 2 недели — замерить сигнал (false positive/negative, скорость alert → подача), затем добавить Слой 2 (1×/день)
4. Слой 3 / career-page fallback — только если предыдущие слои показали ROI

*Явно не делать:* continuous auto-discovery, two-way sync с Notion, попытки парсить JS-SPA без Playwright на этом этапе, обещание 100% покрытия.

#### 2. ResumeBuilder ↔ DB интеграция + status history (объединение функционала с ResumeBuilder)
Полная интеграция ручных заявок (ResumeBuilder workflow) в Postgres + история смены статусов.

**Обновлено 2026-07-01 — пользователь поднял приоритет этого пункта до #2 в роадмапе.** Мотивация: подача резюме идёт именно через ResumeBuilder, но сейчас функция создания карточки в Notion **дублируется** — есть отдельная логика в `notion_client.py` (JobScraper, для найденных скрапером вакансий) и отдельная в ResumeBuilder workflow (для ручных заявок). Точный способ объединения пока не определён — пользователь сознательно фиксирует только направление, не решение.

Возможные направления для объединения (не решено, требует отдельного обсуждения/дизайна перед реализацией):
- Общий модуль создания Notion-карточки, переиспользуемый обоими проектами (единая функция `_make_properties`/create-card логика вместо двух копий)
- ResumeBuilder вызывает JobScraper API/модуль вместо своей копии кода
- Единая точка входа для "создать карточку вакансии" независимо от того, откуда пришла вакансия (скрапер или ручная подача)

**Переосмыслено 2026-07-01** по идее пользователя "Notion должен стать скорее фронтендом, все операции из Notion — дублироваться в Postgres, чтобы отслеживать качество вакансий, успех подачи, какое резюме подавалось, когда ответил рекрутер, как прошло с компанией".

**Обсуждено с продактом и архитектором — вывод: это ~90% тот же самый уже спроектированный пункт ниже, не новая архитектура.** Расхождение — в двух конкретных вещах:

- **Архитектор:** "мониторить все изменения из Notion" не требует ничего сверх уже спланированного `sync_notion.py` (polling по `last_edited_time`) + `status_log`. У Notion нет вебхуков/audit-log на бесплатном тарифе — polling с той же частотой, что и cron скрапера, полностью достаточен (рекрутер отвечает раз в дни, не раз в минуты — "потерянные промежуточные переходы" не имеют значения для этих данных). **Важно: НЕ строить two-way sync** — Notion остаётся UI, Postgres только читает из Notion, никогда не пишет туда в ответ. Единственное реально новое поле — `resume_version`, и оно НЕ может прийти из Notion (там его никогда не было), должно писаться напрямую в `db_manual.py` в момент подачи, когда ResumeBuilder точно знает версию резюме
- **Продакт:** из "качество вакансий / успех подачи / когда ответил рекрутер / как прошло с компанией" реальный ROI при масштабе одного человека — только у **`resume_version`** (какое резюме подавалось — must ship, дёшево, 1 поле) и **outcome/причина** (текстовое поле "почему отклонили/как прошло" — must ship). Timing ответа рекрутера — nice-to-have, не критично при таком объёме заявок. Полная синхронизация email-переписки — explicitly skip, ROI около нуля при таком масштабе

**Итоговая архитектура (уточнена):**

Schema changes:
- `jobs`: добавить `notion_id TEXT UNIQUE`, `current_status TEXT`, `source_type TEXT DEFAULT 'scraper'`, `deleted_at TIMESTAMP`; `run_id` → nullable
- Новый индекс `idx_status_log_job_id ON status_log(job_id)`
- Новая таблица `status_log (id, job_id, old_status, new_status, changed_at, source)`
- Новая таблица `sync_meta (key, value)` — хранит `last_sync_at` для инкрементального polling

- `jobs`: доп. поле `resume_version TEXT` — пишется только из `db_manual.py` в момент подачи (ResumeBuilder знает версию), никогда не читается из Notion
- Статус-vocabulary: явный enum в `config.py`, мапящий значения Notion Status → канонические (`applied → recruiter_reply → screen → interview → offer → rejected`) — без него `status_log` не сможет дать осмысленный "recruiter response timing"

Новые файлы:
- `db_manual.py` — INSERT manual job + `resume_version` сразу после Step 2.5 (merge по url если вакансия уже есть от скрапера)
- `sync_notion.py` — polling Notion с фильтром `last_edited_time >= last_sync_at` (та же частота, что и cron скрапера — этого достаточно, рекрутер отвечает раз в дни, не в минуты), обновляет `current_status`, пишет в `status_log`

Изменения в существующих файлах:
- `db.py` — все JOIN к `runs` → LEFT JOIN
- Step 2.5 в ResumeBuilder CLAUDE.md — добавить вызов `db_manual.py` после создания Notion карточки (fire-and-forget, не блокирует PDF)

Явно НЕ строить:
- Two-way sync / Notion webhooks — Postgres только читает из Notion, никогда не пишет обратно
- `recruiter_replied_at`/outcome как отдельные колонки — это derived-view поверх таймстампов `status_log`, не новые поля
- Синхронизация email-переписки, имени/адреса того кто ответил — ROI около нуля при масштабе одного человека

Known limitations:
- Intermediate status transitions могут теряться между синками (принято как known gap, не критично при частоте изменений раз в дни)
- Notion page deletion: tombstone через `deleted_at` при `archived: true` в API ответе

Pre-ship checklist: идемпотентность upsert, sync failure не блокирует PDF, 3+ реальных статус-переходов end-to-end, Notion "Status" property существует (sync от неё зависит)

Дисциплина использования (продакт): без консистентного ручного логирования статуса вся аналитика ломается — стоит добавить трение в ResumeBuilder workflow (например, смена статуса на "Rejected" в Notion требует явно указать причину — см. "Категории причин отклонения" выше)

Сиквенс относительно остального роадмапа (продакт): EdTech/LMS-источник и эта интеграция могут идти параллельно; "Категории причин отклонения" — сразу после того как status sync заработает (использует ту же инфраструктуру); per-channel дашборд и дайджест — ниже приоритетом

Открытый вопрос: наличие DATABASE_URL в `.env` локально (для вызова из ResumeBuilder контекста)

### Средний приоритет

#### Поддержка приватных (invite-only) Telegram-каналов
Текущий `sources/telegram_channels.py` умеет читать только **публичные** каналы через `t.me/s/{username}` — это не работает для приватных каналов без юзернейма (только invite-ссылка `t.me/+...`).

**Найден кандидат (2026-07-01):** канал "Digital nomads. Work from anywhere" (GoRemote, `t.me/+0zv_MNh22Xw3NTMy`, 13 677 подписчиков, "100% remote jobs — carefully selected from LinkedIn, Indeed, and Telegram"). Проверено вживую — публичного @username нет, только кнопка "JOIN CHANNEL", веб-превью недоступно.

Что нужно для поддержки: реальная Telegram User API сессия (telethon, `TELEGRAM_API_ID`/`TELEGRAM_API_HASH` уже есть в `.env`, сейчас не используются), которая вступает в канал по invite-ссылке и читает сообщения через Client API — принципиально другой код-путь, не расширение текущего HTML-скрапера. Отдельная фича, не просто "добавить канал в список".

Не начинать без явного решения — стоит оценить, сколько ещё таких приватных каналов реально нужно, прежде чем городить второй механизм чтения Telegram.

#### WeWorkRemotely и Arbeitnow: fetch full JD
Аналогично RemoteOK — после обогащения URL загружать полный JD из прямого источника.
Сейчас `fetch_jd_from_url` вызывается только для RemoteOK.

#### Еженедельный дайджест
Вместо 4 уведомлений в день — один дайджест в пятницу.
Топ-5 за неделю, статистика прогонов, сравнение с прошлой неделей.

#### Источник вакансии как отдельное поле в Notion
Сейчас `job["source"]` собирается и хранится в Postgres (`db.py`), но в Notion виден только текстом внутри callout-строки (`ATS Score | Domain | Зарплата | source`).
Добавить отдельное свойство `Source` (select) в Notion DB — чтобы можно было фильтровать/сортировать вакансии по источнику (RemoteOK, Himalayas, Telegram: @channel и т.д.) прямо в таблице.

#### Категории причин отклонения вакансии (manual quality tracking)
Сейчас есть только автоматические причины отклонения (роль/локация/дата/язык — фильтры перед показом). Нужна ещё категоризация **после** того как вакансия дошла до пользователя — чтобы отслеживать качество скрапера и на него влиять.

Причины (select-поле в Notion, заполняется вручную при отклонении вакансии пользователем):
- Плохо подошла по скорингу после проверки — ATS оценил высоко, но по факту не подходит (сигнал на калибровку ATS-рубрики)
- Не глобально-удалённая — локационный фильтр пропустил, а по факту есть региональные ограничения
- Вакансия неактивна — позиция уже закрыта/устарела на момент просмотра (сигнал на слишком долгий MAX_JOB_AGE_DAYS или неактуальные источники)
- Вакансия плохая в принципе — мало информации, невозможно податься, плохая компания

Даёт возможность в дашборде считать долю каждой причины по времени/источнику — например, если "неактивна" стабильно высокая доля у конкретного источника, сокращать для него MAX_JOB_AGE_DAYS или чинить fetch. Пересекается с "Навыковый gap-анализ" ниже — оба используют одну и ту же механику разметки отклонений.

#### Дашборд: список Telegram-каналов с количеством вакансий
Частично уже есть: график "Source performance" в `dashboard.py` группирует по `jobs.source`, а у Telegram-вакансий `source = "Telegram:{channel}"` — то есть каждый канал уже технически отдельная строка. Но:
- Смешано в одном графике со всеми остальными источниками (RemoteOK, Himalayas и т.д.) — при 20 каналах график нечитаем, нужен отдельный раздел/таблица только по Telegram
- Считаются только вакансии, дошедшие до ATS-скоринга (`jobs` таблица хранит только `qualified`/`low_score`). Сырые fetched-до-фильтров числа по каждому каналу отдельно не видны — `scraper.py` сейчас считает `source_counts["TelegramChannels"]` одной суммой на все каналы, а не по каждому

Что сделать:
1. Отдельная таблица/секция в дашборде: канал → всего сообщений спарсено → прошло role-фильтр → qualified (используя уже существующий `jobs.source`)
2. Чтобы увидеть картину *до* фильтров — поменять `scraper.py`, чтобы `telegram_channels.fetch()` возвращал per-channel счётчики, а не только общий список job dict'ов (сейчас в `source_counts` попадает один агрегат)
