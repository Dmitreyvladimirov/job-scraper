# Frontend Design Brief — JobScraper Review UI

> Для дизайн-агента. Дата: 2026-07-15. Решение по scope — `tasks.md` TASK-022,
> `CONTEXT.md` "Divergences from spec".

## Назначение

Соло-инструмент для одного пользователя (Dimitry, поиск PM-вакансий). Расширение
существующего `dashboard.py` (FastAPI + Chart.js, уже на Railway, read-only) до
read-write UI для двух сценариев:

1. **Ручной пересмотр карточек, найденных скрапером** — сейчас происходит в
   Notion, переезжает сюда.
2. **Канбан-движение по статусам воронки** (отклик → ответ рекрутера →
   собеседования → оффер/отказ) + **обязательная причина при отклонении**.

**Явно НЕ в scope:** ручной ввод новых вакансий по URL (это отдельный сервис
`JobPostBot`, остаётся как есть — решение 2026-07-15, см. `tasks.md`), two-way
sync с Notion, email-трекинг переписки с рекрутерами.

## Пользователь

Один человек, использует с телефона и с ноутбука. Не нужна multi-user auth —
достаточно текущей token-based защиты (`?token=`), как в `dashboard.py` сейчас.

## Визуальный референс

Скриншот текущего `dashboard.py` (снят 2026-07-15): `reference/dashboard_current_top.png`
— KPI-плашки, "Qualified jobs per day", "Filter funnel", "ATS score distribution".
Дизайн-агенту стоит открыть его как основу для нового визуального языка (см.
ниже) — не копировать 1:1, но держать как отправную точку.

**Побочная находка со скриншота нижней части дашборда** (не для дизайна, для
бэклога): таблица "Top companies found" показывает **69 записей с пустым
именем компании** и **27 записей, где вместо названия попал текст
cookie-баннера LinkedIn** ("LinkedIn By clicking Continue to join or...",
"respects your privacy LinkedIn and 3rd pa..."). Это не единичный случай (как
казалось раньше по одному примеру в shortlist) — масштаб приличный, ~96 из
записей в этой таблице испорчены. Источник — вероятно `telegram_channels.py`
или generic-URL-скрапинг подтягивает текст со страницы LinkedIn auth-wall в
поле company. Стоит завести отдельным пунктом в `ROADMAP.md`/`tasks.md`,
не в рамках этого брифа.

## Визуальный язык — продолжить существующий, не редизайнить с нуля

`dashboard.py` уже задаёт стиль: тёмная тема, минималистичные карточки,
Chart.js для графиков. Ориентиры из текущей CSS:

- Фон страницы: `#0f1117`; фон карточек/секций: `#1c1f2b`
- Акцент (успех/qualified): `#4ade80` (зелёный)
- Текст: светло-серый на тёмном, приглушённые заголовки (`#888`, uppercase,
  letter-spacing для секций)
- Скругления `10px` на карточках, компактные табличные строки
- Шрифт: системный стек (`-apple-system, BlinkMacSystemFont, 'Segoe UI'`)

Новые экраны должны выглядеть как часть того же продукта, не отдельное
приложение.

## Технические рамки (важно для дизайна — не рисовать несовместимое)

- **Server-rendered HTML + HTMX**, не SPA/React. Взаимодействия — частичные
  HTML-фрагменты по запросу (drag-and-drop канбана, инлайн-смена статуса),
  без полного client-side framework.
- Один FastAPI-сервис (расширение `dashboard.py`), один деплой.
- Мобильная адаптация нужна (телефон — частый сценарий просмотра), но это не
  отдельное native/PWA приложение.

## Экраны

### 1. Card Review (пересмотр находок скрапера)

Список/лента карточек вакансий, попавших в Notion автоматически (замена
текущего просмотра в Notion). На карточку:

- Title, компания, ATS score (число + цветовой индикатор по диапазону)
- Разбивка score: Role / Domain (Value+Exp) / Keywords / Location + penalty
  (сейчас это текстовый callout в Notion — здесь можно визуализировать,
  например мини-барами)
- Matched / missed keywords
- Домен (AI/ML, FinTech, EdTech...), зарплата если есть, источник (RemoteOK,
  Telegram:@channel и т.п.)
- ⚠️ Предупреждения: cooldown (недавно откликался в эту компанию), 🇷🇺 Russia
  warning, ⚠️ incomplete description (короткое AI-саммари вместо полного JD)
- Действия: перевести в воронку ("Подаюсь") / отклонить (см. форма причины
  ниже) / открыть исходную вакансию (ссылка)

### 2. Kanban — движение по статусам воронки

Статусы (канонический словарь, см. `design.md`):
```
found → applied → recruiter_reply → screen → interview → offer
                                                        └→ rejected
```
Карточка перетаскивается между колонками (или тап на мобильном — drag-and-drop
неудобен на телефоне, нужна альтернатива типа dropdown/swipe-action).

**Обязательное поле при переводе в `rejected`** — причина отклонения (см.
ниже), форма/модалка блокирует сохранение без выбора причины.

### 3. Форма причины отклонения

Select с 6 категориями (5 из ROADMAP.md + 6-я добавлена 2026-07-15, различает
ручное решение человека от авто-классификации LLM):

1. Плохо подошла по скорингу после проверки — ATS оценил высоко, по факту не
   подходит
2. Ремоут, но только в одной стране — заявлена как удалённая, региональное
   ограничение (ручное решение пользователя)
3. Вакансия не удалённая вообще — локационный фильтр пропустил, нужен офис
4. Вакансия неактивна / закрыта
5. Вакансия плохая в принципе — мало информации, невозможно податься, плохая
   компания
6. **GEO_RESTRICTED (авто-LLM)** — гео определено автоматически LLM-пайплайном
   как неподходящее (не ручной выбор человека; используется когда карточку
   отклоняет пайплайн, не пользователь напрямую — см. сегодняшний прецедент в
   ResumeBuilder-пайплайне, 84 карточки отклонены так 2026-07-15)

Причина 6 обычно проставляется программно (не через эту форму), но должна
быть видна в UI как read-only тег на карточке, чтобы отличать от причин 1-5.

### 4. Статистика (расширение уже существующих графиков `dashboard.py`)

Новые срезы поверх текущих KPI/графиков:
- Доля каждой причины отклонения по времени / по источнику
- % qualified → applied (метрика качества работы скрапера, не только
  fetch-объём)
- Per-Telegram-channel разбивка (уже отдельно запланировано, `TASK-019`)

## Данные на карточке (источники полей)

Схема ниже полностью зафиксирована в `SPEC_FRONTEND.md` (Phase 1, готова к
Phase 2/3) — для дизайн-агента это означает: поля и их типы больше не
предмет обсуждения, можно проектировать интерфейс под них как под данность.

| Поле | Источник |
|---|---|
| Title, Company | `jobs.title`, `jobs.company` (Postgres, существует) |
| ATS score + breakdown | `jobs.ats_score`, `jobs.domain` + разбивка по 4 осям (Role/Domain/Keywords/Location) — уже существует |
| why_apply | `jobs.why_apply TEXT` (новое поле, `SPEC_FRONTEND.md`) |
| why_not | `jobs.why_not` (существует) |
| Matched/missed keywords | `jobs.matched_keywords`, `jobs.missed_keywords` — `JSONB`, массив строк (новые поля, `SPEC_FRONTEND.md`) |
| Source | `jobs.source` (существует) |
| Apply URL | `jobs.apply_url` (существует) |
| Status (воронка) | `jobs.current_status TEXT DEFAULT 'found'` (новое поле, `SPEC_FRONTEND.md` — независимо от REQ-111/112, тот трек по-прежнему не реализован и не блокирует это) |
| Rejection reason | `jobs.rejection_reason TEXT`, 6 категорий из раздела выше (новое поле, `SPEC_FRONTEND.md`) |
| Cooldown / Russia warning | вычисляется на лету (`notion_client.py` logic) или кэшируется — не менялось |

**Известное ограничение (не блокирует дизайн, важно для реализации):**
старые вакансии, найденные ДО деплоя этой фичи, не будут иметь
`why_apply`/`matched_keywords`/`missed_keywords` (поля `NULL`) — досчитывать
задним числом решено не делать (подтверждено пользователем 2026-07-15).
Дизайн-агенту стоит предусмотреть, как карточка выглядит с пустыми
keywords/why_apply (не только "богатое" состояние).

- Как визуализировать 4-осевой ATS breakdown (бары/радар/просто числа)
