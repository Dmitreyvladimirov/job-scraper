# Job Scraper — Техническое задание

## Назначение

Автоматизированный бот для поиска PM-вакансий на международных job boards. Бот запускается 4 раза в день, фильтрует нерелевантные вакансии, оценивает соответствие резюме, сохраняет результаты в Notion и отправляет сводку в Telegram.

---

## Стек

| Компонент | Технология |
|---|---|
| Язык | Python 3.12+ |
| Хостинг | Railway (cron job) |
| Хранилище вакансий | Notion Database |
| Хранилище статистики | PostgreSQL (Railway managed Postgres) |
| LLM | OpenAI GPT-4o-mini |
| Уведомления | Telegram Bot API |
| Обогащение URL | Greenhouse / Lever / Ashby Public API |


### Telegram каналы (sources/telegram_channels.py)

Скрапит публичные страницы `https://t.me/s/{channel}`. Авторизация не нужна.

**Список каналов:** `TELEGRAM_JOB_CHANNELS` в `.env` (comma-separated).

**URL-классификация сообщения:**
1. Приоритет 1 — ссылка с лейблом "тут / apply / откликнуться"
2. Приоритет 2 — известный ATS-домен (Greenhouse, Lever, Ashby, Workable, Remocate...)
3. Приоритет 3 — первая ссылка в первых 300 символах (Remocate-стиль)
4. Fallback — t.me ссылка на само сообщение

**Listing pages:** URL вида `/careers` или `/jobs` без конкретного job ID распознаётся как листинг. Скрапер пытается открыть страницу и найти внутри отдельные PM-вакансии. Если страница недоступна (403, JS-SPA) — fallback на t.me ссылку.

**Russia detection:** если в location/company/тексте сообщения/описании найден российский город или ключевое слово "россия/russia" — выставляется `russia_warning = True`.

**Извлечение текста сообщения:** HTML каждого сообщения парсится через `<br>` → `\n` замену, затем `get_text()` без разделителя (исправлено 2026-07-02; раньше `get_text(separator="\n")` вставлял перенос строки на границе КАЖДОГО тега, не только `<br>`, что разбивало "Title в **Company**" на две "строки" и ломало извлечение компании — см. известные ограничения ниже).

**linkedin.com полностью исключён** из фетчащихся/apply-URL (весь домен в `_SKIP_URL_PATTERNS`, исправлено 2026-07-02) — любая страница LinkedIn (посты, джоб-листинги) без логина отдаёt один и тот же auth-wall текст ("Sign in to LinkedIn... By clicking Continue to join or sign in..."), который раньше протекал в поле "Компания".

## Pipeline обработки вакансии

```
Fetch all sources
    │
    ▼
Strip HTML из descriptions
    │
    ▼
Cross-source dedup (company + title)
  — если дубли: берём не-RemoteOK, потом длиннейшее описание
    │
    ▼
[Для каждой вакансии]
    │
    ├─ passes_role_filter?       — title содержит PM-ключевые слова
    ├─ passes_language_filter?   — English / Spanish / Russian (исключение: требует русский)
    ├─ passes_location_filter?   — Remote worldwide / Israel / EMEA
    ├─ passes_date_filter?       — не старше MAX_JOB_AGE_DAYS (14 дней)
    ├─ URL dedup (seen_urls)?    — не было в Notion по URL
    ├─ Key dedup (seen_keys)?    — не было в Notion по (company, title)
    │
    ▼
enrich_url()                     — для RemoteOK/Arbeitnow/WWR: найти прямую ссылку
    │                              через Greenhouse → Lever → Ashby API по названию компании
    ▼
[Telegram] fetch_jd_from_url() или fetch_url_generic()  — полный JD по ссылке из сообщения
[RemoteOK] fetch_jd_from_url() → fetch_url_generic()    — полный JD; apply_url из API или найденный выше
  (если ни один не дал текст → флаг incomplete_description, callout в Notion)
    │
    ▼
ats.analyze()                    — GPT оценивает вакансию (≤ MAX_GPT_CALLS_PER_RUN = 40)
    │
    ├─ score < 60 → create_rejected_entry() в Notion (не виден в основном фильтре)
    └─ score ≥ 60 → create_entry() в Notion + добавить в top_jobs
    │
    ▼
telegram.send_run_summary()      — итоги прогона + топ вакансии
db.finish_run()                  — сохранить статистику в SQLite
```

## ATS-скоринг (ats.py)

Модель: GPT-4o-mini, temperature=0. Лимиты контекста: JD — первые 5000 символов, резюме — первые 7000 символов.

Четыре измерения, максимум 100 баллов:

| Измерение | Макс | Описание |
|---|---|---|
| Role Match | 30 | Senior PM/Head/VP = 25–30; mid PM = 12–22; PO/Associate = 0–10 |
| Domain Fit | 30 | Value (0–15) + Experience (0–15) раздельно |
| Keyword Overlap | 25 | Must Have (2 балла) + Nice to Have (1 балл) по 12 ключевым словам |
| Location | 15 | Remote/Israel = 15; EMEA = 8; US only = 0 |
| **Penalty** | −15 | Если JD требует N лет специфического домена которого нет |

Порог для Notion qualified: **60+**. Ниже — `rejected_by_scraper`.

Domain Value шкала: AI/ML=14–15, B2B SaaS=12–14, Cybersecurity=11–12, FinTech=10–11, EdTech=9–11, Data/BI=7–9, Growth/B2C=5–7.

## URL-обогащение (utils.py)

### RemoteOK — правило обогащения

**Правило:** для каждой вакансии из RemoteOK скрапер ищет прямую ссылку на вакансию в компании на внешних ATS-платформах.

Порядок действий:
1. `enrich_url(job)` — генерирует slug-варианты из названия компании, ищет через:
   - Greenhouse API (`boards-api.greenhouse.io`)
   - Lever API (`api.lever.co`)
   - Ashby GraphQL (`jobs.ashbyhq.com`)
   - Валидирует через `_company_match()` (защита от "Insider" → "Business Insider")
2. Если нашёл → `job["apply_url"]` = прямая ссылка
3. Если не нашёл через ATS → берём `apply_url` из RemoteOK API (прямая ссылка на сайт компании)
4. По найденному URL: `fetch_jd_from_url()` (ATS API) → fallback `fetch_url_generic()` (HTML scraping)
5. Если JD получить не удалось → `incomplete_description = True`, callout в Notion `⚠️`

### Все источники
`fetch_jd_from_url()` умеет парсить Greenhouse / Lever / Ashby через их API (возвращают чистый текст).
`fetch_url_generic()` — универсальный HTML scraper для всех остальных страниц (strip script/style/nav).

## Notion-карточка (notion_client.py)

Поля в карточке:
- `Позиция` — "Role Title (Company)"
- `Компания` — отдельное поле (plain text)
- `Ссылка на вакансию` — прямая ссылка если нашли, иначе платформа
- `ATS Score` — число 0–100 (сортировка/фильтрация)
- `Status2` — "Scraped" для новых, "rejected_by_scraper" для низкого скора
- `Статус` — "Активно"
- `Date Applied` — дата создания карточки

Тело страницы: callout с ATS breakdown (Role/Domain/Keywords/Location + Value·Exp детализация), `why_apply`, `why_not`, matched/missed keywords. При cooldown и неполном описании — дополнительные callout-предупреждения.

**Russia warning:** если `russia_warning = True` — добавляется callout 🇷🇺 и поле `Тип вакансии = "🇷🇺 Russia"`. В Telegram-сводке такая вакансия помечается флагом 🇷🇺.

**UTF-16:** Notion считает длину текста в UTF-16 code units (как JavaScript). Emoji вне BMP (🌎🚀 и др.) занимают 2 единицы. Все rich_text блоки нарезаются через `_split_for_notion()` с лимитом 1990 UTF-16 единиц (не символов Python).

## Дедупликация

**Слой 1 — внутри прогона (cross-source):**
Группируем по `normalize_job_key(company, title)`. Из дублей выбираем лучший источник: не-RemoteOK приоритетнее, затем длиннейшее описание.

**Слой 2 — между прогонами (Notion + Postgres):**
При старте загружаем из Notion все URL (`seen_urls`) и все пары (company, title) (`seen_keys`). Postgres также хранит историю прогонов. Вакансия пропускается если совпадает по любому из двух.

`normalize_job_key()` стрипает суффиксы (Inc, Ltd, GmbH, LLC...) и пунктуацию для fuzzy-матчинга.

**Важно:** `seen_urls.add()` вызывается всегда после обработки вакансии, независимо от успеха записи в Notion. Это предотвращает дубли внутри одного прогона (в предыдущей версии URL добавлялся только при успехе → один и тот же URL мог попасть в Notion несколько раз если первые попытки падали с ошибкой).

## Фильтры

### Языковой фильтр
Пропускаются: **английский, испанский, русский**.
Блокируются: немецкий, французский, нидерландский и другие.

Детекция:
1. Хард-блок: маркеры `(m/w/d)`, `:in` суффикс (немецкая гендерно-нейтральная форма)
2. Мягкий блок: 3+ стоп-слова из списка немецкого/французского/нидерландского

Исключение: любая вакансия с явным требованием русского языка проходит фильтр.

### Cooldown компаний
Если в Notion уже есть отклик на эту компанию за последние 90 дней — вакансия создаётся с предупреждением в callout, но не блокируется.

## Известные проблемы и ограничения (актуально на 2026-07-02)

Раздел для QA — что тестировать как "уже должно работать" и что тестировать как "известно, что сломано, не дублировать баг-репорт".

### Уже исправлено (2026-07-02)
- LinkedIn как источник JD/apply-URL полностью исключён (весь домен, не только `/posts/`) — раньше давал auth-wall текст вместо описания вакансии.
- Regex бэкфилла компании из JD (`scraper.py`) сужен до явных лейблов "company"/"компания"/"работодатель" — раньше триггерился на словах "at"/"in" где угодно в тексте, включая LinkedIn-заглушку.
- `get_text(separator="\n")` в парсинге Telegram-сообщений заменён на `<br>`→`\n` + `get_text()` без разделителя — раньше разбивал "Title в **Company**" на разные "строки", если компания выделена жирным/курсивом/ссылкой (частый случай), и терял компанию.

### Открытые баги (см. BACKLOG.md для деталей)
- **`_company_match()` не применяется к Lever и Ashby** (`utils.py::find_apply_url()`) — только Greenhouse-ветка защищена от ложных совпадений компании по slug. Риск: apply_url может вести на вакансию другой компании.
- **Извлечение title/company из Telegram-сообщений (`_extract_title_company()` в `sources/telegram_channels.py`) смотрит только на первую строку текста.** Известные паттерны, которые оно НЕ обрабатывает:
  - Сообщения со списком из нескольких ролей + компания на отдельной строке ниже (напр. канал `forproducts`: 18 релевантных вакансий за прогон, **0% с извлечённой компанией**)
  - Явно лейблированный формат "🏢 Company: X" на отдельной строке (канал `remotejobss`) — при этом берётся ПЕРВАЯ строка как title (обычно generic-заголовок типа "JOB OPPORTUNITY"), из-за чего role-фильтр отбрасывает **вообще все** сообщения канала — на practике канал сейчас даёт 0 вакансий несмотря на чистый, легко парсящийся формат
  - Суффикс "(Компания)" в самом title не отделяется (напр. `smartremotejobs`, `productjobgo`) — title = "Senior PM (CyberNut)", company остаётся пустой
  - Некоторые каналы (`workewco`) вообще не упоминают компанию в тексте сообщения — не баг, ограничение источника
- **Два Telegram-канала из списка (`agile_jobs`, `cryptojobswork`) не отдают список сообщений через `t.me/s/` вообще** (публичное веб-превью отключено владельцем) — требует Telegram Client API (telethon), не HTML-скрапинг. Канал `jobstobefound` формально жив, но не постил ничего последние ~3.5 месяца (не баг).
- Полный аудит по всем 21 каналу — см. историю сессии 2026-07-02 или запросить повторно у Claude.

### GitHub Actions manual trigger сломан
`.github/workflows/scraper.yml` (`workflow_dispatch`) не передаёт `DATABASE_URL` в шаг "Run scraper" — при ручном запуске `db.init_db()` упадёт с `EnvironmentError`. Нужно добавить секрет и env var перед использованием.

# ТЗ на следующую версию (v2)

Консолидация `ROADMAP.md` (что планируется) + `BACKLOG.md` (что не так сейчас) в формате, пригодном для тест-плана. Порядок — по приоритету. Каждый пункт: что меняется, зачем, как проверить, что НЕ входит в scope.

## Высокий приоритет

### v2.1 — Фикс `_company_match()` на Lever/Ashby
**Сейчас:** `utils.py::find_apply_url()` защищает от ложного совпадения компании по slug (напр. "Insider" → "Business Insider") только на Greenhouse-ветке. Lever и Ashby принимают первый ответивший slug без проверки имени компании.
**Как проверить:** взять вакансию, у которой slug компании неоднозначен (общее слово), убедиться что `apply_url` ведёт на вакансию ИМЕННО этой компании, а не однофамильца на Lever/Ashby.
**Не входит:** join.com (проверяется отдельно, см. v2.3).

### v2.2 — Telegram: чинить конкретные каналы, найденные аудитом
**Сейчас:** `forproducts` (крупнейший источник релевантных вакансий из Telegram) теряет 100% компаний из-за multi-bullet формата; `remotejobss` теряет 100% вакансий из-за неверного title (берётся первая строка-заголовок вместо реального названия роли на второй строке).
**Как проверить:** прогнать `_fetch_channel('forproducts', ...)` и `_fetch_channel('remotejobss', ...)` — компания/title должны совпадать с тем, что реально написано в сообщении (не пустое, не generic "JOB OPPORTUNITY").
**Не входит:** `agile_jobs`/`cryptojobswork` (не отдают контент через `t.me/s/` вообще — требует telethon, см. v2.6), суффикс "(Компания)" в title (мельче, ниже приоритетом).

### v2.3 — Прямые источники по компаниям (EdTech/LMS company-direct)
**Сейчас:** такого источника нет вообще — компании опрашиваются только если попали в один из аггрегаторов/Telegram случайно.
**Что появится:** новый источник `sources/company_direct.py`, опрашивающий Greenhouse/Lever/Ashby/join.com напрямую по списку ~80 целевых EdTech/LMS компаний (после чистки — см. ROADMAP §1). Три слоя по частоте: Слой 1 (израильские компании + 8 крупных глобальных, 4×/день, instant Telegram-алерт при находке), Слой 2 (остальные крупные глобальные, 1×/день), Слой 3 (fallback на существующие источники). Плюс generic-скрапер карьерных страниц для компаний без известного ATS (server-rendered страницы; JS-SPA не покрывается).
**Как проверить:** после реализации — компания из Слоя 1 с новой вакансией на Greenhouse/Lever/Ashby должна появиться в Notion **до** следующего батч-прогона (мгновенный алерт), а не только в общей сводке 4×/день. Таблица `target_companies` в Postgres должна отражать статус (active/pending/dormant) каждой компании.
**Не входит:** continuous auto-discovery новых компаний (ручной quarterly review), 100%-покрытие (реалистично ~53% через структурированный ATS).

### v2.4 — GitHub Actions manual trigger
**Сейчас:** `workflow_dispatch` в `.github/workflows/scraper.yml` не передаёт `DATABASE_URL` → падает с `EnvironmentError` при ручном запуске.

## Средний приоритет

### v2.5 — ResumeBuilder ↔ Postgres синхронизация статусов
**Сейчас:** ручные заявки (через ResumeBuilder) и их статусы (ответил рекрутер / скрин / оффер / отказ) существуют только в Notion, не попадают в Postgres → нет истории, нет аналитики по конверсии.
**Что появится:** `db_manual.py` (INSERT вручную поданной вакансии + поле `resume_version`, которое ResumeBuilder знает в момент подачи), `sync_notion.py` (одностороннее чтение статусов из Notion по `last_edited_time`, пишет в новую таблицу `status_log`). Строго одностороннее — Postgres никогда не пишет обратно в Notion.
**Как проверить:** после ручной подачи через ResumeBuilder — запись должна появиться в Postgres `jobs` с `source_type='manual'` и `resume_version` заполненным. Смена статуса в Notion (например Applied → Screen) должна отразиться в `status_log` после следующего sync-цикла.
**Не входит:** two-way sync, синхронизация email-переписки, авто-заполнение timing ответа рекрутера как отдельного поля (это derived-view поверх `status_log`, не хранится отдельно).

### v2.6 — Поддержка приватных Telegram-каналов
**Сейчас:** только публичные каналы через `t.me/s/{username}`. Приватные (invite-only, `t.me/+...`) не читаются вообще.
**Что появится:** telethon-based клиент (используя уже существующие, но неактивные `TELEGRAM_API_ID`/`TELEGRAM_API_HASH`), вступающий в канал по invite-ссылке и читающий через Client API — отдельный код-путь, не расширение HTML-скрапера.
